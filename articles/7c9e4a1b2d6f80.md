最近はAIにコードを書いてもらう機会が増え、出力されたコードを理解し直す場面も多くなりました。
そのなかで、AIがときどき `==` を使うことがありました。私はこれまで何となく `==` を避けていましたが、「なぜ避けるのか」を明確に説明できないことに気づきました。
そこで本記事では、JavaScript における `==` の仕様と、開発での使い分けを整理します。AIが出力したコードに対しても、根拠を持って判断できるようになることを目指します。
今回は MDN / ECMAScript仕様 の内容をもとに `==` の理解を深めていきます。

## `==` と `===` の違い

`===` は厳密等価演算子と呼ばれます。こちらについては皆さんご存知の通り、基本的に型と値の両方が一致するときのみ true を返します。

```js
1 === 1;     // true
1 === "1";   // false
true === 1;  // false
```

一方、`==`は等価演算子と呼ばれます。

```js
1 == "1";    // true: "1"が数値の1に変換される
true == 1;   // true: trueが数値の1に変換される
false == 0;  // true: falseが数値の0に変換される
```

私は、`==` は左右の値が falsy かどうかを基準に判定しているのだと思っていました。しかし、実際にはそうではありません。
例えば、次の式については true になります。

```js
[] == false; // true
```

直感的には、何も問題ないように見えます。

しかし、[MDN](https://developer.mozilla.org/ja/docs/Glossary/Falsy) によると、falsy な値は次の通りです。
`null` / `undefined` / `false` / `NaN` / `0` / `-0` / `0n` / `""` / `document.all`
この一覧に `[]` は含まれていません。

なので、truthy == falsy が true になっています。こう見ると直感的ではなくなります。
試しに、`[]`を Boolean へ変換してから比較すると `false` になります。

```js
Boolean([]); // true: 空配列はtruthy
[] == false; // true
Boolean([]) == false; // false
```

もし falsy かどうかを比べているのであれば、truthy な空配列と `false` が等しくなることはありません。
`==` の[仕様（MDN）](https://developer.mozilla.org/ja/docs/Web/JavaScript/Reference/Operators/Equality#%E8%A7%A3%E8%AA%AC)には、左右の型が異なる場合に型変換することがあると明記されています。この型変換によって、このような不思議な（？）挙動になっています。

## 抽象等価比較

ECMAScript仕様では、`==` による比較を[抽象等価比較（IsLooselyEqual）](https://tc39.es/ecma262/multipage/abstract-operations.html#sec-islooselyequal)という処理で定義しています。

仕様の詳細を初めからすべて追うのは大変なので、まずは次の流れを押さえるところから理解していきます。

1. 左右が同じ型なら、その型の規則で比較する
2. `null` と `undefined` の組み合わせなら `true` にする
3. 片方が数値、もう片方が文字列なら、文字列を数値に変換する
4. 片方が真偽値なら、真偽値を数値に変換する
5. 片方がオブジェクト、もう片方がプリミティブ値なら、オブジェクトをプリミティブ値に変換する
6. 必要なら、変換後の値でもう一度比較する

このほかに `bigint` が関わる規則もありますが、これは後述します。

ここでいうプリミティブ値は、オブジェクトではない値です。`string`、`number`、`bigint`、`boolean`、`undefined`、`symbol`、`null`が該当します。

### 同じ型同士の比較

左右が同じ型の場合、`==` は `===` とまったく同じ結果になります。仕様上も、同じ型同士の比較は厳密等価比較（IsStrictlyEqual）へ委ねられています。

```js
"hello" == "hello"; // true
10 == 10;           // true
10 == 20;           // false
```

オブジェクト同士では、内容ではなく参照が同じかを比較します。

```js
const a = { name: "Iririn", age: 18 };
const b = { name: "Iririn", age: 18 };

a == b; // false
a == a; // true
```

オブジェクトリテラルは、評価されるたびに新しいオブジェクトを生成します。

`a` と `b` には、それぞれ別のオブジェクトへの参照が代入されます。そのため、プロパティと値が同じでも `a == b` は `false` になります。

一方、`a == a` は同じオブジェクトへの参照を比較するため `true` になります。

### `null` と `undefined` の比較

冒頭の「『無い』もの同士を勝手に仲良くさせないでよ！」の部分です。
[仕様（IsLooselyEqualの手順2・3）][is-loosely-equal]では、片方が `null` でもう片方が `undefined` の場合、そのまま `true` を返すと定義されています。

つまり、`null == undefined` が `true` になるのは型変換の結果ではなく、仕様に直接書かれた特別ルールです。

[is-loosely-equal]: https://tc39.es/ecma262/multipage/abstract-operations.html#sec-islooselyequal

```js
null == undefined; // true
```

ここで重要なのは、`null` と `undefined` が `==` で等しくなる相手は「自分自身とお互いだけ」という点です。
`Number(null)` は `0` になりますが、`==` の比較で `null` や `undefined` が数値に変換されることはありません。

```js
Number(null); // 0

null == 0;      // false
null == false;  // false
null == "";     // false
undefined == 0; // false
```

この性質を利用したのが、`x == null` というイディオムです。

```js
if (x == null) {
  // x が null または undefined のときだけここに入る
}
```

これは `x === null || x === undefined` と同じ意味になります。`0` や `""` などの falsy な値を巻き込まないため、「値が無いこと」だけを確認したい場面では `==` が役立つ、例外的なケースとして知られています。

#### linter ルールも `== null` を例外扱いしている

このイディオムが広く認められている証拠に、`==` を禁止する linter ルール自体が `== null` だけを例外扱いするオプションを持っています。

ESLint の [eqeqeq](https://eslint.org/docs/latest/rules/eqeqeq) ルールには、`null` との比較にだけルールを適用しない設定があります。

```json
{ "eqeqeq": ["error", "always", { "null": "ignore" }] }
```

`"smart"` オプションでも `foo == null` は正しいコードの例として挙げられています。

Biome の [noDoubleEquals](https://biomejs.dev/linter/rules/no-double-equals/) では、
デフォルトで `== null` を許容しています（`ignoreNull` オプションの初期値が `true`）。
ドキュメントにも for ergonomic reasons と、利便性のための意図的な例外であることが
明記されています。

### 数値と文字列の比較

冒頭の「0 == "" が `true` ってどういうこと？」の部分です。
[仕様（IsLooselyEqualの手順4・5）][is-loosely-equal]では、片方が数値でもう片方が文字列の場合、文字列を `ToNumber` で数値に変換してから比較すると定義されています。

```js
0 == "";   // true: "" が 0 に変換される
0 == "0";  // true: "0" が 0 に変換される
1 == "1";  // true: "1" が 1 に変換される
```

`ToNumber` による文字列の変換は、`Number()` 関数と同じ規則です。空文字列や空白だけの文字列は `0` になり、数値として解釈できない文字列は `NaN` になります。

```js
Number("");    // 0
Number("  ");  // 0
Number("0");   // 0
Number("abc"); // NaN
```

`0 == ""` が `true` になるのは、`""` が `0` に変換され、`0 == 0` として比較されるためです。「空文字列と `0` が等しい」というより、「空文字列を数値化したら `0` だった」という方が実態に近いです。

なお、`NaN` はどんな値とも等しくならないため、数値化できない文字列との比較は必ず `false` になります。

```js
0 == "abc"; // false: "abc" が NaN に変換され、0 == NaN は false
```

#### `"" == "0"` が `false` になる理由

冒頭の「0 == "" は `true` なのに、"" == "0" は `false` って何なの！？」もここで回収できます。

`"" == "0"` は文字列同士の比較です。「同じ型同士の比較」の章で見た通り、型変換は発生せず、文字列として内容を比較します。`""` と `"0"` は内容が異なるので `false` です。

```js
0 == "";    // true:  "" → 0 に変換して比較
0 == "0";   // true:  "0" → 0 に変換して比較
"" == "0";  // false: 同じ型なので変換なし、文字列として比較
```

つまり、A == B と B == C が `true` でも A == C が `true` になるとは限りません。`==` の等価性は推移的でなく、これが `==` を避けるべきとされる大きな理由のひとつです。

### 真偽値の比較

[仕様（IsLooselyEqualの手順8・9）][is-loosely-equal]では、片方が真偽値の場合、真偽値を `ToNumber` で数値に変換してから比較をやり直すと定義されています。`true` は `1` に、`false` は `0` に変換されます。

```js
true == 1;  // true:  true が 1 に変換される
false == 0; // true:  false が 0 に変換される
true == 0;  // false: 1 == 0 になる
```

ここで注目すべきは、変換されるのが真偽値の側だという点です。相手側の値が真偽値に変換されるわけではありません。この方向性が、直感に反する結果を生みます。

```js
true == 2;      // false: true が 1 に変換され、1 == 2
true == "true"; // false
```

`2` は truthy な値ですが、`true == 2` は `false` です。
`2` が `true` に変換されるのではなく、`true` が `1` に変換され、`1 == 2` として比較されるためです。
`true == "true"` も同様に、`"true"` が真偽値として解釈されることはありません。

また、相手が文字列の場合は変換が連鎖します。

```js
true == "1"; // true
// 1. true が 1 に変換され、1 == "1" になる
// 2. 「数値と文字列の比較」の規則で "1" が 1 に変換され、1 == 1 になる
// 3. true
```

まず真偽値の規則で `true` が `1` に変換され、次に「数値と文字列の比較」の規則で `"1"` が `1` に変換されます。仕様の手順が「変換して終わり」ではなく「変換してから比較をやり直す」形になっているため、片方の変換が終わるたびに、残りの規則が再び適用されていきます。

`false` 側でも同じことが起きます。

```js
false == "";  // true: false → 0、"" → 0 で 0 == 0
false == "0"; // true: false → 0、"0" → 0 で 0 == 0
```

#### `x == true` は「truthy かどうか」の判定にならない

この規則から導かれる重要な注意点があります。`x == true `と書いても、「x が truthy かどうか」の判定にはなりません。

```js
if (2) {
  // 2 は truthy なのでここに入る
}

2 == true;   // false: true が 1 に変換され、2 == 1
"a" == true; // false: true → 1、"a" → NaN で 1 == NaN
```

`if (2)` は実行されるのに、`2 == true` は `false` です。truthy かどうかを判定したい場合は、`x == true` ではなく `if (x)` や `Boolean(x)` を使います。

なお、「`==` と `===` の違い」の章で見た `[] == false` も、まずこの規則で `false` が `0` に変換され、`[] == 0` になります。残りの `[]` の変換については、次のオブジェクトとプリミティブ値の比較で説明します。

### オブジェクトとプリミティブ値の比較

冒頭の「[] == ![] が `true` って何！？」の部分です。
[仕様のオブジェクトとプリミティブ値を比較する規則][is-loosely-equal]では、
片方がオブジェクトでもう片方が数値・文字列・BigInt・Symbol の場合の規則が定義されています。
オブジェクトを `ToPrimitive` でプリミティブ値に変換してから、比較をやり直します。

`ToPrimitive` は、おおまかに次の順序でオブジェクトをプリミティブ値へ変換します。

1. `[Symbol.toPrimitive]` メソッドがあれば、`"default"` をヒントとして呼び、結果がプリミティブ値ならそれを使う
2. なければ `valueOf` を呼び、結果がプリミティブ値ならそれを使う
3. それでもプリミティブ値にならなければ `toString` を呼び、結果がプリミティブ値ならそれを使う

メソッドがプリミティブ値を返さないままこの手順が終わると、`TypeError` が発生します。

配列やオブジェクトリテラルで作ったオブジェクトは `[Symbol.toPrimitive]` を持たず、`valueOf` は自分自身（オブジェクト）を返します。
そのため、実質的に `toString` の結果が使われます。
Dateなど、`[Symbol.toPrimitive]` を持つオブジェクトでは異なる変換結果になることがあります。

```js
[].toString();     // ""
[1].toString();    // "1"
[1, 2].toString(); // "1,2"
({}).toString();   // "[object Object]"
```

これを踏まえると、次の結果が説明できます。

```js
[] == "";   // true: [] → "" に変換され、"" == ""
[] == 0;    // true: [] → ""、さらに "" → 0 で 0 == 0
[1] == 1;   // true: [1] → "1"、さらに "1" → 1 で 1 == 1
({}) == "[object Object]"; // true
```

ここでも「真偽値の比較」と同じく変換が連鎖します。オブジェクトが文字列になったあと、相手が数値なら「数値と文字列の比較」の規則が再び適用されます。

#### `[] == ![]` が `true` になる理由

冒頭の「自分の否定と等しいってどういう存在なの！？」をここで回収します。

ポイントは、`!` が `==` より先に評価されることです。
`[]` は truthy なので、`![]` は比較が始まる前にただの `false` になっています。
つまり `[] == ![]` は、「配列と自分の否定の比較」ではなく、実際にはこれまで見てきた `[] == false` そのものです。

```js
[] == ![];
// 1. ![] が先に評価される。[] は truthy なので ![] は false
// 2. [] == false になる
// 3. 「真偽値の比較」の規則で false → 0 になり、[] == 0
// 4. この章の規則で [] → "" になり、"" == 0
// 5. 「数値と文字列の比較」の規則で "" → 0 になり、0 == 0
// 6. true
```

3つの規則をまたいで変換が3回連鎖した結果、最終的に `0 == 0` が比較されていました。哲学的な存在に見えたものの正体は、地道な型変換の積み重ねです。

#### オブジェクトと `null` ・ `undefined` の比較は `false`

この規則で変換が起きる相手は、数値・文字列・BigInt・Symbol だけです。`null` と `undefined` はこの一覧に含まれないため、オブジェクトとの比較ではどの規則にも該当せず、仕様の最後の「 `false` を返す」に到達します。

```js
[] == null;       // false
[] == undefined;  // false
({}) == null;     // false
```

これは「`null` と `undefined` の比較」の章で見た `x == null` イディオムの裏付けにもなっています。
次に説明する `document.all` を除き、`x` がオブジェクトなら `x == null` は `false` になるため、通常は「値が無いこと」の判定に使えます。

なお、ブラウザー環境の `document.all` だけは歴史的経緯（[Annex B][is-html-dda]）により、オブジェクトなのに `== null` が `true` になる唯一の例外です。
冒頭の falsy 一覧に `document.all` が紛れ込んでいたのは、このためです。

[is-html-dda]: https://tc39.es/ecma262/multipage/additional-ecmascript-features-for-web-browsers.html#sec-IsHTMLDDA-internal-slot-aec

### BigIntの比較

「抽象等価比較」の章で後述するとしていた、`bigint` が関わる規則です。
冒頭の falsy 一覧には `0n` も紛れ込んでいました。
BigInt にも `==` の変換規則がふたつ定義されています。

#### BigIntと文字列の比較

[仕様（IsLooselyEqualの手順6・7）][is-loosely-equal]では、BigInt と文字列を比較する場合、
文字列を `StringToBigInt` で BigInt に変換してから比較をやり直すと定義されています。

```js
1n == "1"; // true: "1" が 1n に変換される
0n == "";  // true: "" が 0n に変換される
```

ここで使われるのは、「数値と文字列の比較」で登場した `ToNumber` ではなく `StringToBigInt` です。BigInt の整数リテラルとして解釈できない文字列は、`NaN` ではなく `undefined` になります（`BigInt()` 関数なら例外を投げるケースです）。変換結果が `undefined` の場合、仕様はその場で `false` を返します。

```js
1 == "1.0";  // true:  ToNumber("1.0") は 1
1n == "1.0"; // false: 小数点は BigInt の構文にないので変換できない
1n == "1n";  // false: 末尾の n も構文には含まれない
```

同じ「数値っぽい文字列」でも、比較相手が `number` か `bigint` かで結果が変わります。

#### BigIntと数値の比較

[仕様（IsLooselyEqualの手順12）][is-loosely-equal]では、BigInt と数値を比較する場合の規則が定義されています。
ここでは珍しく、どちらの側も型変換しません。

1. どちらかが `NaN`・`Infinity`・`-Infinity` なら `false` を返す
2. 両者の数学的値（mathematical value）が等しければ `true`、異なれば `false` を返す

```js
1n == 1;        // true
1n == 1.5;      // false
1n == NaN;      // false
1n == Infinity; // false
```

数学的値で比較するのは、型変換による精度の喪失を避けるためです。
Number は 2^53 を超えると整数を正確に表せません。
もし BigInt を Number に変換してから比較する仕様だったら、本当は異なる値どうしが等しいと判定されてしまいます。

```js
const big = 9007199254740993n; // 2n ** 53n + 1n

Number(big); // 9007199254740992: Number に変換すると精度が落ちる
big == 9007199254740992; // false: 変換せず数学的値どうしで比較するため
```

#### 他の規則との連鎖

BigInt が相手でも、これまでの章と同じように変換は連鎖します。

```js
1n == true; // true
// 1. 「真偽値の比較」の規則で true → 1 になり、1n == 1
// 2. 「BigIntと数値の比較」の規則で、数学的値が等しいので true

[1n] == 1n; // true
// 1. 「オブジェクトとプリミティブ値の比較」の規則で [1n] → "1" になり、"1" == 1n
// 2. 「BigIntと文字列の比較」の規則で "1" → 1n になり、1n == 1n
// 3. 同じ型なので厳密等価比較になり true
```

なお、`===` は型が異なれば必ず `false` なので、`1n === 1` は `false` です。
BigInt と Number をまたいで値の一致を確かめたい場面は、`==` が素直に役立つ数少ないケースといえます
（もっとも、`BigInt(x) === y` と明示的に変換した方が意図は伝わります）。

## 開発での使い分け

ここまで見てきた通り、`==` の挙動はすべて仕様に書かれた規則の積み重ねです。
とはいえ、「説明できる」ことと「使いやすい」ことは別の話です。
最後に、開発でどう使い分けるかを整理します。

### 基本は `===` を使う

原則はシンプルで、迷ったら `===` です。

`==` を避ける最大の理由は、読み手に高いコストを強いる点にあります。
ここまで見てきた通り、`==` の変換規則は型の組み合わせごとに存在し、しかも連鎖します。
`[] == ![]` を読み解くだけで、3つの規則をまたぐ必要がありました。
コードを読むたびに、この手順を頭の中で再生し続けるわけにもいきません。

また、「数値と文字列の比較」の章で見た通り、`==` は推移的な等価性を持ちません。
一方、型が同じ値同士なら `==` と `===` の結果は変わらないため、
`===` に統一しても失うものはほとんどありません。

### 例外は `x == null` と BigInt

すでに見たふたつの例外を再掲します。

「`null` と `undefined` の比較」の章で見た `x == null` は、
`null` と `undefined` をまとめて判定できる実用的なイディオムです。
linter も例外として認めており、チームで合意できていれば採用する価値があります。
より明示的に書きたい場合は、`x === null || x === undefined` と展開します。
あるいは、`x ?? defaultValue` のように Null 合体演算子で「値が無い」ケースを吸収する方法もあります。

「BigIntの比較」の章で見た BigInt と Number の比較も、
`==` が数学的値で正しく判定してくれる場面でした。
ただし、こちらは `BigInt(x) === y` のような明示的な変換で置き換えられます。

### NaNの判定には `===` も `==` も使えない

冒頭の「NaN === NaN が `false` って何なの？ 自分自身と等しくない数って哲学なの！？」を、最後に回収します。

[仕様（Number::equal）][number-equal]では、比較のどちらかが `NaN` なら `false` を返すと明記されています。
`===`（IsStrictlyEqual）も、同じ型同士の `==` も、数値の比較は最終的にこの規則へ行き着きます。

```js
NaN === NaN; // false
NaN == NaN;  // false
```

これは JavaScript の独自仕様ではなく、浮動小数点数の国際標準である IEEE 754 に由来します。
`NaN` は `0 / 0` や `Math.sqrt(-1)` のような、数値として定義できない計算の結果を表す値です。
出どころの異なる「失敗した計算の結果」同士を等しいとみなすわけにはいかないため、
`NaN` はどの値とも（自分自身とも）等しくないと定義されています。
哲学に見えたものの正体は、標準規格でした。

[number-equal]: https://tc39.es/ecma262/multipage/ecmascript-data-types-and-values.html#sec-numeric-types-number-equal

開発で `NaN` かどうかを判定したいときは、`Number.isNaN()` を使います。

```js
Number.isNaN(NaN);   // true
Number.isNaN("abc"); // false: 数値ではないので NaN そのものではない
```

似た名前のグローバル関数 `isNaN()` には注意が必要です。
こちらは引数を `ToNumber` で数値に変換してから判定するため、
数値化できない値がすべて `true` になります。

```js
isNaN("abc");        // true: ToNumber("abc") が NaN になるため
Number.isNaN("abc"); // false
```

また、`Object.is()` を使うと `NaN` 同士を等しいと判定できます。

```js
Object.is(NaN, NaN); // true
```

`Object.is()` は SameValue という、`===` とも `==` とも異なる3つ目の比較アルゴリズムです。
`NaN` 同士を `true` にするほか、`0` と `-0` を区別します
（`0 === -0` は `true` ですが、`Object.is(0, -0)` は `false` です）。

配列メソッドの検索にも、この違いが現れます。

```js
[NaN].indexOf(NaN);  // -1:   indexOf は === 相当で比較する
[NaN].includes(NaN); // true: includes は NaN 同士を等しいとみなす
```

`includes` は SameValueZero という比較を使います。
`NaN` 同士を等しいとみなし、`0` と `-0` は区別しない、SameValue と `===` の中間のような規則です。

### 判断を linter に任せる

方針が決まったら、人間が覚えておくのではなく linter に任せます。
「`null` と `undefined` の比較」の章で見た通り、linter は `==` の禁止ルールを備えています。
ESLint の eqeqeq や Biome の noDoubleEquals を有効にすれば、
`== null` だけを許可しつつ、それ以外の `==` を機械的に排除できます。
仕様を知らなくても事故は防げますが、仕様を知っていれば、警告の意味まで説明できるようになります。

## まとめ

本記事では、MDN と ECMAScript 仕様をもとに `==` の挙動を整理しました。
冒頭で「わかんない」と叫んでいた疑問は、すべて仕様の規則で説明できます。

- `0 == ""` → `""` が `ToNumber` で `0` になるため `true`
- `"" == "0"` → 同じ型同士は型変換なしで比較されるため `false`
- `null == undefined` → 仕様が直接 `true` と定めている特別ルール
- `[] == ![]` → `![]` が先に `false` と評価され、3つの規則をまたぐ変換の連鎖で `true`
- `NaN === NaN` → IEEE 754 由来で、`NaN` はどの値とも等しくないため `false`

こうして並べると、`==` は決められた手順を淡々と実行しているだけだとわかります。
「ゆるふわ比較」に見えたのは、規則を知らなかったからでした。

その上で、開発時の方針はシンプルです。
基本は `===` を使います。
例外は `x == null` のイディオム、および BigInt と Number をまたぐ比較のふたつです。
`NaN` の判定には `Number.isNaN()` を使います。
そして、これらの方針は linter に任せて機械的に守ります。

これまで何となく `==` を避けていた理由に、ようやく根拠を持てました。
AIが `==` を含むコードを出力してきたときも、「なんとなく危ないから」ではなく、
仕様を基準に採否を判断できます。
